## 简洁、简单且实用

Vivid 的核心理念是在保障 Actor 模型基础能力的前提下，**优先优化开发者体验**，通过极致简洁的设计减少额外认知负荷。这一原则细化为三个维度：

**简洁**：最小化复杂度与依赖
- 严格控制公共 API 的导出范围，仅暴露必要接口，避免过度设计。
- 最小化外部依赖引入，核心库仅依赖经严格筛选的基础包，从源头规避依赖冲突与安全风险。

**简单**：直觉化接口与透明性
- 追求接口的直觉化设计，多数场景下通过单次调用即可完成核心操作。
- 对本地与远程通信的差异保持透明：开发者无需手动区分调用类型，由框架基于 `ActorRef` 自动处理路由与传输。

**实用**：开放扩展与灵活定制
- 通过定义清晰的接口暴露扩展点（如编解码器 `Codec`、邮箱 `Mailbox`、桥接器 `Bridge` 等），允许开发者按需替换或增强核心组件。
- 避免“一刀切”的实现，支持根据业务场景定制监督策略、调度逻辑等非通用能力。

## 消息、序列化

消息是 Actor 模型的通信核心，Vivid 对消息定义、序列化及传输的设计遵循**安全优先、透明传输、灵活扩展**原则：

**消息定义**：类型安全与最小暴露
- 为开发者可处理的消息提供**强类型定义**，确保编译期类型校验。
- 避免过度暴露内部实现细节，仅通过必要接口提供消息发送能力。

**序列化**：自定义优先与兜底保障
- 支持开发者通过实现 `Codec` 接口自定义消息序列化逻辑，适配 Protobuf、JSON 等任意格式。
- 对未自定义序列化的消息，框架提供**内置安全兜底机制**，尽可能的确保序列化/反序列化可行性。
- 任何序列化/反序列化失败均通过**结构化错误**结合事件、日志明确反馈，拒绝静默失败。

**消息传输**：位置透明的自动化处理
- 基于 `ActorRef` 的统一寻址模型：发送方仅需持有目标 `ActorRef`，无需关心其物理位置（本地/远程）。
- 框架自动完成路由决策、序列化、网络传输及反序列化，开发者全程无需介入底层细节。

## 清晰的错误处理

Vivid 确保所有错误的**可判定性**与**可观测性**，严格遵循 Go 错误处理最佳实践：

- **标准化接口**：所有框架错误实现 `error` 接口，支持 `Is`（错误匹配）、`As`（类型断言）、`Unwrap`（错误链展开）方法。
- **结构化错误码**：框架错误通过预定义错误码（如 `ErrActorNotFound`、`ErrSerializationFailed`）标识类型，文档集中说明各码含义。
- **分布式一致性**：错误信息（含错误码、堆栈）在跨节点传播时无损保留，支持远程节点与本地一致的错误处理逻辑。
- **错误链透明**：复杂操作的错误通过链式包装（如 `fmt.Errorf("send failed: %w", err)`）保留上下文，便于根因定位。

## 性能标准

Vivid 认同性能是系统基石，但在多数场景下，**易用性、简洁性与开发者体验**具有更高优先级。核心取舍逻辑为：**当性能损耗处于可接受范围时，优先保障用户体验**。

### 案例分析：`tell` 函数的设计
传统 Actor 框架可能要求开发者手动区分本地/远程发送（如 `tellLocal`/`tellRemote`），但 Vivid 选择统一接口：

- **传统做法的代价**：  
  - 增加认知负担：开发者需实时判断目标 Actor 位置；  
  - 破坏 API 简洁性：函数碎片化导致调用逻辑冗余；  
  - 阻碍无缝扩展：单机到分布式的迁移需修改大量调用点。  

- **Vivid 的方案**：  
  提供统一 `tell(actorRef ActorRef, msg interface{})` 函数。发送方通过 `ActorRef` 包含的地址信息（本地/远程标识），由框架自动完成路由与传输，全程透明。

在此场景下，区分本地/远程调用的性能损耗**微乎其微**（纳秒级路由判断开销），却为开发者带来**显著体验提升**：消除手动选择成本、确保 API 一致性、奠定无缝扩展基础。性能优化应聚焦于瓶颈（如序列化、网络 IO），而非此类表层差异。
